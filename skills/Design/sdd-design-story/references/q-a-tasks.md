## 场景识别

在开始问答之前，识别 story 的类型，选择合适的问答策略。

### 场景类型

**1. 业务需求场景**
- **特征**：复杂的功能开发，需要详细的任务分解
- **示例**：新增用户管理、新增支付功能、新增报表功能
- **问答策略**：询问任务粒度、任务依赖、测试策略

**2. 设计分解场景**
- **特征**：任务框架已在设计阶段明确，需要澄清具体任务
- **示例**：实现数据访问层、实现 API 网关、实现消息队列
- **问答策略**：跳过"任务策略"，聚焦具体任务细节和实施步骤

**3. 技术改进场景**
- **特征**：任务相对简单，聚焦验证和回滚
- **示例**：优化查询性能、重构认证模块、升级依赖版本
- **问答策略**：聚焦改进步骤、验证方法、回滚方案

### 场景识别逻辑

**判断依据**（基于步骤 1 的场景识别）：
- 如果步骤 1 识别为"业务需求" → 使用【业务需求场景】问答逻辑
- 如果步骤 1 识别为"设计分解" → 使用【设计分解场景】问答逻辑
- 如果步骤 1 识别为"技术改进" → 使用【技术改进场景】问答逻辑

## 问答流程逻辑

### 业务需求场景

#### 初始阶段（第一轮）

**目标**：识别任务分解策略

**必问问题**：
1. **任务粒度偏好**
   - "任务拆分的粒度偏好是什么？"
   - "细粒度（每个任务 1-2 小时）/中粒度（每个任务 2-4 小时）/粗粒度（每个任务 4-8 小时）？"

2. **开发优先级**
   - "哪些模块需要优先完成？"
   - "核心业务逻辑/用户界面/数据存储/测试用例？"

**输出格式**：
```markdown
📊 当前步骤：步骤 4 - 开发任务规划（业务需求场景）
🎯 问答进度：初始阶段（0/4 维度已覆盖）

📋 任务分解策略：
- 任务粒度：[细粒度/中粒度/粗粒度]
- 优先级：[优先完成的模块]

❓ 请回答以下问题：

1. 您希望任务拆分的粒度是什么？（细粒度/中粒度/粗粒度）
2. 哪些模块需要优先完成？
```

#### 深化阶段（第二轮）

**目标**：深入任务依赖和测试策略

**问题选择策略**：

**如果有数据模型变更** → 询问**数据层任务**
- "数据迁移脚本如何编写？"
- "是否需要回滚脚本？"

**如果有多个模块** → 询问**任务依赖**
- "任务之间的依赖关系是什么？"
- "哪些任务可以并行开发？"

**如果有复杂业务逻辑** → 询问**测试策略**
- "单元测试覆盖哪些场景？"
- "集成测试如何设计？"

**默认深化问题**：
1. **代码提交**
   - "每个任务对应一次提交吗？"
   - "提交消息符合什么规范？"

2. **风险点**
   - "哪些任务技术风险较高？需要预研吗？"

**输出格式**：
```markdown
📊 当前步骤：步骤 4 - 开发任务规划（业务需求场景）
🎯 问答进度：深化阶段（2/4 维度已覆盖）

✅ 已覆盖维度：
- 任务粒度偏好
- 开发优先级

❓ 请回答以下问题：

1. 任务之间的依赖关系是什么？哪些可以并行？
2. 单元测试覆盖哪些场景？集成测试如何设计？
```

#### 确认阶段（最后轮）

**目标**：确认任务列表和执行计划

**必问问题**：
1. **任务完整性**
   - "是否有遗漏的任务？"
   - "是否需要补充辅助任务（如文档更新、环境配置）？"

2. **执行计划**
   - "任务的总工时是多少？"
   - "是否需要分阶段交付？"

**理解确认**：
```markdown
📝 我理解的任务分解：

**任务粒度**：
- [总结任务粒度]

**开发优先级**：
- [总结开发优先级]

**任务依赖关系**：
- [总结任务依赖关系]

请确认理解是否正确：
- [ ] 正确，继续下一步
- [ ] 需要修正
```

## 边界自检

在进入下一轮问答前，执行以下边界自检：

### 自检清单

- [ ] 是否在重新讨论业务规则？（如"这个规则是否合理？"）
  - 如果是 → 提醒用户返回步骤2（增量规格）修改
- [ ] 是否在重新讨论技术方案？（如"这个技术选型是否合适？"）
  - 如果是 → 提醒用户返回步骤3（增量设计）修改
- [ ] 任务是否具体到文件/模块级别？
  - 如果否 → 继续细化任务描述
- [ ] 任务粒度是否合理（1-4小时可完成）？
  - 如果否 → 继续拆分或合并任务

### 边界提醒话术

如果用户开始讨论业务规则或技术方案，使用以下话术：

```markdown

当前我们正在讨论 Tasks（任务规划）：
- 业务规则已在步骤2（增量规格）中定义

如果您想修改业务规则或技术方案，我们可以：

- [ ] 返回步骤3（增量设计）修改技术方案
```

- [ ] 已将技术方案拆分为可执行的任务
- [ ] 任务粒度合理（1-4小时可完成）
- [ ] 任务描述具体到文件/模块级别
- [ ] 任务依赖关系清晰
- [ ] 每个任务都有明确的验证标准
- [ ] 每个业务规则至少有一个测试用例
- [ ] 任务引用了delta-spec.md和delta-design.md
- [ ] 识别了高风险任务并有缓解方案
