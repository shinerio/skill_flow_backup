## CodeSpec 元论

### 核心理念

规格驱动开发（SDD）的核心理念是：**先想清楚，再动手写代码**。

### 为什么 SDD？

传统开发的痛点：
1. **需求不清晰**：细节在开发中才暴露，导致频繁返工
2. **设计与实现脱节**：代码结构混乱，缺少架构决策记录
3. **缺少可追溯性**：需求 → 规格 → 设计 → 代码 的链接不明确

SDD 的解决方案：
1. **结构化思考**：通过 5 个阶段强制思考过程
2. **文档驱动**：每个阶段都有明确的输出文档
3. **可追溯性**：文档链提供完整的决策历史
4. **增量开发**：ADDED/MODIFIED/REMOVED 语义追踪变更

### 设计原则（来自 CodeSpec）

1. **单一真理源** — 业务逻辑在代码仓库中仅有一份权威定义
2. **去官僚化** — 消除冗余元数据，拒绝双重记账
3. **人机共生** — 文档优先服务人类认知，标准化格式利于 AI 验证

## 阶段 1: 需求澄清

### 认知目标
将模糊的需求转化为清晰的需求证据。

### 为什么要有这个阶段？
在 CodeSpec 工作流中，proposal.md 是 AR 的起点。它回答的核心问题是：
- **我们为什么要做这个？**（Why - 动机）
- **我们要做什么？**（What Changes - 变更内容）

### 思考框架

1. **这个需求解决的核心问题是什么？**
   - 问题的本质是什么？
   - 谁遇到了这个问题？
   - 问题的严重程度如何？

2. **谁会获得价值？**
   - 识别利益相关者
   - 对每个利益相关者分别说明价值

3. **如果只能做 50%，最核心的部分是什么？**
   - 优先级框架：P0（MVP 必须）、P1（重要增强）、P2（未来优化）

4. **不做的是什么（不在范围内）？**
   - 明确排除的功能点，避免范围蔓延

### 质量标准

- **问题清晰度**：能否用一句话描述"我们解决的是什么问题"？
- **范围边界明确**："做什么"和"不做什么"的边界是否清晰？
- **可设计性**：是否提供了足够的技术设计输入？
- **可测量性**：是否定义了功能点、里程碑、完成标志？

### 不要做什么

- ❌ 不要讨论技术实现细节（这是 Design 阶段的事）
- ❌ 不要过早拆分成任务（这是 Tasks 阶段的事）
- ❌ 不要试图一次性写出完美的需求描述（迭代是正常的）

## 阶段 2: Spec增量设计

### 认知目标
将需求证据转化为可验收的业务规则增量。

### 为什么要有这个阶段？
proposal.md 描述了"做什么"（What Changes），但没有回答"业务规则是什么"（What & Why）。
delta-spec.md 的核心任务是：
- **新增的业务规则是什么（ADDED）？**
- **哪些规则需要修改（MODIFIED）？**
- **哪些规则需要删除（REMOVED）？**

### 思考框架

1. **哪些业务规则是全新的（ADDED）？**
   - 从 proposal.md 的功能清单中，提取对应的业务规则
   - 格式："触发场景 → 预期行为"

2. **哪些规则需要修改（MODIFIED）？**
   - 对照 SPEC.md 基线，识别需要修改的现有规则
   - 说明修改原因："原来的规则不适应新需求"

3. **哪些规则需要删除（REMOVED）？**
   - 识别被废弃的功能
   - 解释删除原因和迁移路径

4. **每个规则的验收条件是什么？**
   - 必须是可测试的、可验证的条件

5. **Spec vs Design 的边界在哪里？**
   - Spec：订单ID必须全局唯一
   - Design：订单ID使用雪花算法，存储为BIGINT
   - 重要：不要在 Spec 中讨论技术实现细节

### 质量标准

- **ADDED**：新规则是否覆盖了 proposal.md 的所有功能点？
- **MODIFIED**：修改是否基于 SPEC.md 的现有规则，而非重新定义？
- **REMOVED**：删除是否有明确的原因和归档计划？
- **可验证性**：每条规则是否都有明确的验收条件？
- **边界清晰**：是否保持了 Spec vs Design 的边界？

### Spec vs Design 的边界示例

| 描述 | 属于 Spec | 属于 Design |
|------|-----------|-------------|
| 订单ID必须全局唯一 | ✅ | |
| 订单ID使用雪花算法 | | ✅ |
| 商品数量必须为正整数，上限50 | ✅ | |
| 商品列表使用JSON字段存储 | | ✅ |
| 金额精度为小数点后两位 | ✅ | |
| 金额使用DECIMAL(18,2)类型 | | ✅ |

**重要**：在 delta-spec.md 中只描述 Spec 相关的内容，不要讨论技术实现细节。

## 阶段 3: Design增量设计

### 认知目标
将业务规则转化为可实施的技术方案。

### 为什么要有这个阶段？
delta-spec.md 描述了"业务规则是什么"（What & Why），但没有回答"如何实现"（How）。
delta-design.md 的核心任务是：
- **用什么技术实现？**（技术选型、方案对比）
- **数据模型如何设计？**（与现有模型如何关联）
- **接口如何设计？**（参数、返回值、错误处理）
- **有什么权衡取舍？**（风险与缓解、非目标）

### 思考框架

1. **技术选型**：用什么技术实现？
   - 有哪些备选方案？
   - 方案对比（优点、缺点）
   - 选择理由

2. **数据模型**：数据结构如何设计？
   - 新增表：字段、类型、约束、索引
   - 修改表：新增字段、修改字段、DDL 语句
   - 与现有模型的兼容性

3. **接口设计**：API 如何设计？
   - 新增接口：路径、请求体、响应体、错误码
   - 修改接口：变更说明、新增参数、向后兼容性

4. **权衡取舍**：有什么考虑？
   - 性能 vs 可维护性
   - 简单 vs 灵活
   - 风险与缓解

5. **非目标**：不做的是什么？
   - 明确不涉及的内容，避免范围蔓延

### 质量标准

- **技术选型**：是否有方案对比和选择理由？
- **数据模型**：是否考虑了与现有模型的兼容性？
- **接口设计**：是否向后兼容（如需修改）？
- **风险评估**：是否识别了关键风险和缓解方案？
- **可实施性**：DDL 语句是否准备好执行？
- **可追溯性**：技术方案是否与 delta-spec.md 的业务规则保持一致？

### 不要做什么

- ❌ 不要重复描述业务规则（这在 delta-spec.md 中）
- ❌ 不要过早讨论具体的代码实现（这是 Tasks 阶段的事）

## 阶段 4: 任务拆解

### 认知目标
将技术方案拆分为 AI Agent 可执行的任务。

### 为什么要有这个阶段？
delta-design.md 描述了"技术方案是什么"（How），但没有回答"具体怎么实现"。
tasks.md 的核心任务是为 AI Agent（如 Claude Code / Cursor）提供可执行的任务清单。

### 思考框架

1. **任务粒度**：任务拆分是否合适？
   - 建议每个任务 1-4 小时可完成
   - 如果任务太大，继续拆分

2. **依赖关系**：任务之间的依赖是否清晰？
   - 数据模型 → 领域层 → 应用层 → 基础设施层 → 接口层 → 测试
   - 避免循环依赖

3. **具体性**：任务是否具体到文件/模块级别？
   - 任务描述应明确包含文件路径、方法名、具体操作
   - 示例："创建服务类：`src/application/services/OrderService.ts`"

4. **可验证性**：每个任务是否可独立验证？

### 质量标准

- **单一职责**：每个任务专注一个文件或模块
- **可执行性**：任务描述是否具体到文件/模块级别？
- **验证标准**：是否定义了"怎么算完成"？
- **测试覆盖**：每个业务规则是否至少有一个测试用例？
- **可追溯性**：任务是否引用 delta-spec.md 和 delta-design.md？

### AI Agent 执行指引

1. **按顺序执行**：遵循依赖关系
2. **参考文档**：
   - 业务规则参考：delta-spec.md
   - 技术设计参考：delta-design.md
   - 全量规格参考：SPEC.md
   - 全量设计参考：DESIGN.md
3. **代码规范**：遵循项目现有代码风格
4. **测试覆盖**：每个业务规则至少有一个测试用例
5. **提交粒度**：每完成一个模块可以单独提交

### 模板参考

详见 [tasks.md 模板](../../../../../../Onebox/CloudBU-doc/2026.01.01%20SDD/codespec-ar-workflow/references/templates/tasks.md)。

## 跨阶段的质量标准

### 追溯性

完整的需求 → 规格的设计。追溯到功能点：
- proposal.md 的"功能清单"是否都能追溯到 delta-spec.md 的"业务规则"？
- delta-spec.md 的"业务规则"是否都能追溯到 delta-design.md 的"技术方案"？
- delta-design.md 的"技术方案"是否都能追溯到 tasks.md 的"任务"？

### 一致性

**Spec 与 Design 的一致**：
- delta-spec.md 的 Spec vs Design 边界是否清晰？
- delta-design.md 的技术方案是否与 delta-spec.md 的业务规则一致？

**全量与增量的一致**：
- delta-doc 的变更是否与全量文档（SPEC.md / DESIGN.md）兼容？
- 合并检查清单是否完整？

### 完整性

每个阶段是否覆盖了 CodeSpec 标准模板的所有章节？

**proposal.md**：
- 背景与动机（现状痛点、业务驱动）
- 变更内容（功能清单、用户故事、不在范围内）
- 影响分析（受影响的规格、受影响的设计、破坏性变更、依赖关系）
- DFX约束（性能、可靠性）
- 里程碑

**delta-spec.md**：
- ADDED Requirements（业务规则、交互流程、异常场景）
- MODIFIED Requirements（修改的现有业务规则）
- REMOVED Requirements（删除的业务规则）
- 数据约束变更（ADDED/MODIFIED）
- 术语变更（ADDED/MODIFIED）
- 合并检查清单

**delta-design.md**：
- 设计背景（设计目标、设计约束、非目标）
- 设计决策（方案对比、选择理由）
- 数据模型变更（新增表、修改表、DDL语句）
- 接口设计变更（新增接口、修改接口）
- 流程设计（PlantUML图表）
- 风险与缓解
- 待解决问题
- 合并检查清单

**tasks.md**：
- 数据模型（迁移脚本、表创建/修改、索引）
- 领域层（实体、值对象、仓储接口）
- 应用层（服务类、DTO、用例）
- 基础设施层（仓储实现、外部服务客户端）
- 接口层（Controller、验证器、路由）
- 测试（单元测试、集成测试、测试数据）
- 文档更新（API文档、合并增量到基线）
- 验证（运行测试、lint检查、本地验证）
- 任务依赖关系
- AI Agent 执行指引
