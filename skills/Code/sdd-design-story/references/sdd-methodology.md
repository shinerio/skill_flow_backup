# SDD 方法论背景

本文档介绍规格驱动开发（SDD，Specification Driven Design）的核心概念、设计原则和实践方法。

---

## 核心理念

规格驱动开发（SDD）的核心理念是：**先想清楚，再动手写代码**。

---

## 为什么需要 SDD？

### 传统开发的痛点

1. **需求不清晰**：细节在开发中才暴露，导致频繁返工
2. **设计与实现脱节**：代码结构混乱，缺少架构决策记录
3. **缺少可追溯性**：需求 → 规格 → 设计 → 代码 的链接不明确

### SDD 的解决方案

1. **结构化思考**：通过 5 个阶段强制思考过程
2. **文档驱动**：每个阶段都有明确的输出文档
3. **可追溯性**：文档链提供完整的决策历史
4. **增量开发**：ADDED/MODIFIED/REMOVED 语义追踪变更

---

## 设计原则

### 1. 单一真理源

业务逻辑在代码仓库中仅有一份权威定义。

### 2. 去官僚化

消除冗余元数据，拒绝双重记账。

### 3. 人机共生

文档优先服务人类认知，标准化格式利于 AI 验证。

```

(proposal) (delta-spec) (delta-design) (tasks) (validation)
```

### 追溯性

- proposal.md 的"功能清单" → delta-spec.md 的"业务规则"
- delta-spec.md 的"业务规则" → delta-design.md 的"技术方案"
- delta-design.md 的"技术方案" → tasks.md 的"任务"

### 一致性

- Spec 与 Design 边界清晰
- 全量与增量文档兼容
- 合并检查清单完整
