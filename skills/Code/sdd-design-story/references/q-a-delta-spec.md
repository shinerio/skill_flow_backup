# 步骤 2：增量规格定义的问答逻辑

本文档提供 delta-spec.md 生成过程中的验证问题集和流程逻辑，用于澄清规格细节和变更点。

**前置条件**：已读取全量规格文档（`{root}/codespec/specs/spec.md`）

> **通用引导技巧**：详见 [./common-guidance.md](./common-guidance.md)

---

## 🚨 阶段边界提醒

**当前阶段：步骤2 - 增量规格定义（Spec）**

**本阶段聚焦**：
- ✅ 业务规则是什么（What & Why）- ADDED/MODIFIED/REMOVED
- ✅ 约束条件和验收标准
- ✅ 异常场景处理
- ✅ 数据约束（业务层面，如"金额必须为正数"）

**本阶段禁止**：
- ❌ 技术实现细节（如算法、数据类型、存储方式）→ 步骤3（Design）
- ❌ 数据模型设计（如表结构、字段定义）→ 步骤3（Design）
- ❌ 具体接口参数格式（如JSON Schema、参数类型）→ 步骤3（Design）
- ❌ 数据迁移策略（DDL语句、迁移脚本）→ 步骤3（Design）

**技术细节将在步骤3的Design阶段讨论**

## 场景识别

在开始问答之前，识别 story 的类型，选择合适的问答策略。

### 场景类型

**1. 业务需求场景**
- **特征**：新增业务功能，需要定义新的业务规则
- **示例**：新增用户管理、新增支付功能、新增报表功能
- **问答策略**：询问新增的业务规则、约束条件、异常场景

**2. 设计分解场景**
- **特征**：业务规则已在 proposal 中明确，需要澄清细节
- **示例**：实现数据访问层、实现 API 网关、实现消息队列
- **问答策略**：跳过"业务规则是什么"，聚焦规则细节和边界条件

**3. 技术改进场景**
- **特征**：技术优化、重构，无业务规则变更
- **示例**：优化查询性能、重构认证模块、升级依赖版本
- **问答策略**：聚焦是否有隐藏的业务规则影响、兼容性、数据约束

### 场景识别逻辑

**判断依据**（基于步骤 1 的场景识别）：
- 如果步骤 1 识别为"业务需求" → 使用【业务需求场景】问答逻辑
- 如果步骤 1 识别为"设计分解" → 使用【设计分解场景】问答逻辑
- 如果步骤 1 识别为"技术改进" → 使用【技术改进场景】问答逻辑

## 问答流程逻辑

### 业务需求场景

#### 初始阶段（第一轮）

**目标**：识别核心变更点

**必问问题**：
1. **变更识别**
   - "基于 proposal.md，主要变更点有哪些？"
   - "是新增功能、修改现有功能，还是删除功能？"

2. **业务规则概览**
   - "新增的业务规则是什么？"
   - "哪些现有规则需要修改？"

**输出格式**：
```markdown
📊 当前步骤：步骤 2 - 增量规格定义（业务需求场景）
🎯 问答进度：初始阶段（0/4 维度已覆盖）

📋 基于 proposal.md 的变更识别：
- [ ] 新增功能
- [ ] 修改现有功能
- [ ] 删除功能

❓ 请回答以下问题：

1. 主要变更点是什么？请列举关键的业务规则变化。
2. 这些变更会影响哪些现有的业务流程？
```

#### 深化阶段（第二轮）

**目标**：深入业务规则和边界条件

**问题选择策略**：

**如果有新增功能** → 询问**新增规则细节**
- "新增的业务规则有哪些？请逐条描述。"
- "新增规则的约束条件是什么？"

**如果有修改现有功能** → 询问**修改详情**
- "哪些现有规则需要修改？修改点在哪里？"
- "修改后的行为与之前有什么不同？"

**如果有删除功能** → 询问**删除原因**
- "为什么要删除这些规则？"
- "删除后如何处理历史数据？"

**默认深化问题**：
1. **边界条件**
   - "极端情况如何处理？如最大值、最小值、空值。"
   - "异常场景有什么？如何处理？"

2. **状态转换**
   - "涉及哪些状态变化？转换条件是什么？"

**输出格式**：
```markdown
📊 当前步骤：步骤 2 - 增量规格定义（业务需求场景）
🎯 问答进度：深化阶段（2/4 维度已覆盖）

✅ 已覆盖维度：
- 变更识别
- 业务规则概览

❓ 请回答以下问题：

1. 新增的业务规则有哪些？约束条件是什么？
2. 极端情况和异常场景如何处理？
```

#### 确认阶段（最后轮）

**目标**：验证规格完整性和兼容性

**必问问题**：
1. **数据约束**
   - "数据模型有哪些变更？"
   - "字段约束是否有变化？"

2. **兼容性**
   - "新的规格是否向后兼容？"
   - "需要哪些版本迁移措施？"

**理解确认**：
```markdown
📝 我理解的规格变更：

**ADDED Requirements**：
- [总结新增规则]

**MODIFIED Requirements**：
- [总结修改规则]

**REMOVED Requirements**：
- [总结删除规则]

**数据约束变更**：
- [总结数据约束]

请确认理解是否正确：
- [ ] 正确，继续下一步
- [ ] 需要修正
```

## 问题库

### 业务规则

- 具体的业务逻辑是什么？有哪些约束条件？
- 系统行为有什么变化？
- 状态转换逻辑是什么？
- 新增的业务规则有哪些？请逐条描述。
- 哪些现有规则需要修改？修改点在哪里？

### 边界条件

- 极端情况如何处理？错误场景怎么办？
- 异常情况如何处理？
- 输入数据无效时系统应该如何处理？
- 最大值、最小值、空值等情况如何处理？

### 状态转换

- 涉及哪些状态变化？转换条件是什么？
- 状态转换逻辑是什么？
- 是否有循环状态？如何避免？

### 接口定义

- 新增/修改的接口有哪些？
- 接口需要满足什么业务要求？
- 接口错误时如何处理（业务层面）？
- 接口变更对现有调用者的影响？

### 数据约束

- 数据有哪些业务约束？
- 数据验证规则是什么？
- 数据一致性要求是什么？

### 兼容性

- 新的规格是否向后兼容？
- 需要哪些版本迁移措施？
- 数据兼容性如何保证？

## 验收标准检查

在生成 delta-spec.md 之前，检查以下内容：

### ✅ ADDED Requirements

- [ ] 新规则是否覆盖了 proposal.md 的所有功能点？
- [ ] 每条规则是否有明确的约束条件？
- [ ] 是否定义了异常场景的处理方式？

### ✅ MODIFIED Requirements

- [ ] 修改是否基于 SPEC.md 的现有规则，而非重新定义？
- [ ] 修改后的行为是否与 proposal.md 一致？
- [ ] 是否说明了修改的原因和影响？

### ✅ REMOVED Requirements

- [ ] 删除是否有明确的原因？
- [ ] 是否有归档计划？
- [ ] 是否考虑了历史数据的处理？

### ✅ 数据约束变更

- [ ] 字段约束是否明确？
- [ ] 是否定义了数据验证规则？
- [ ] 是否考虑了数据迁移？

### ✅ 边界清晰

- [ ] 是否保持了 Spec vs Design 的边界？
- [ ] 没有讨论技术实现细节（如算法、数据类型）

## Spec vs Design 边界检查

在问答过程中，始终注意避免进入 Design 领域：

### ❌ 不应该问的问题（属于 Design）

- "订单ID使用什么算法生成？"（Design）
- "数据存储在哪个表中？"（Design）
- "使用什么技术实现？"（Design）
- "接口的参数类型是什么？"（Design）

### ✅ 应该问的问题（属于 Spec）

- "订单ID必须满足什么要求？"（Spec）
- "订单ID是否需要全局唯一？"（Spec）
- "订单ID的格式是什么？"（Spec）
- "订单ID的长度限制是多少？"（Spec）

### 边界提醒

如果用户开始讨论技术实现，提醒：
```markdown
💡 提醒：技术实现细节属于 Design 阶段

当前我们正在讨论 Spec（业务规则），技术实现细节（如算法、数据类型、存储方式）会在步骤 3 的 Design 阶段讨论。

如果您想讨论技术方案，我们可以：
- [ ] 先完成 Spec 阶段，再进入 Design 阶段
- [ ] 跳转到步骤 3 的 Design 阶段
```
