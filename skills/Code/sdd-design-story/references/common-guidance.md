# 通用引导技巧

本文档提供适用于所有 SDD 设计步骤的问答技巧，帮助更好地引导用户提供信息。

---

## 基础提问技巧

### 开放式问题

- "你能描述一下..."（引导用户详细说明）
- "你认为最重要的方面是什么？"（引导用户明确优先级）

### 封闭式问题

- "是否需要..."（确认具体需求）
- "是 A 还是 B？"（在选项中做出选择）

### 追问技巧

- "为什么？"（了解背后的原因）
- "具体是什么意思？"（澄清模糊描述）
- "举个例子？"（通过实例说明）

### 确认技巧

- "我理解的是...，对吗？"（确认理解是否正确）
- "所以你的意思是..."（总结并确认）

## 问答分段机制

### 分段原则

**每次提问限制**：
- 每次输出 2-3 个问题，避免信息过载
- 单次回复总行数控制在 30 行以内
- 问题较多时，拆分为多轮对话逐步推进

**问题选择策略**：
- 优先询问必须回答的核心问题
- 根据用户回答动态调整后续问题
- 已回答的问题不再重复询问

### 进度跟踪格式

在对话中维护以下状态信息：

```markdown
**当前步骤**: 步骤 X - [步骤名称]
**问答进度**: [已完成/进行中] (已覆盖维度数/总维度数)
**已覆盖维度**:
- ✅ 业务价值
- ✅ 目标用户
- ⏳ 功能范围（待确认）
- ⏸️ 用户故事（待开始）
```

### 重入时的问答恢复

当 skill 被重新激活时：

1. **检查对话历史**
   - 查找最近的问答进度记录
   - 识别已覆盖的维度和待覆盖的维度

2. **确认理解**
   - 对已回答的维度进行确认："我理解您的核心痛点是...，对吗？"
   - 给用户修正的机会："如果有理解偏差，请告诉我"

3. **继续问答**
   - 优先询问未覆盖的维度
   - 根据用户反馈调整问题顺序

## 上下文状态管理

### 状态跟踪变量

在对话中维护以下关键信息：

```markdown
**状态跟踪**:
- `current_step`: 当前步骤（0-5）
- `work_dir`: 当前工作目录（如 `US202601010015-增加逻辑多租`）
- `qa_progress`: 各步骤问答进度记录
- `generated_docs`: 已生成的文档列表
- `last_action`: 上次执行的操作（问答/生成/跳转）
```

### 状态输出格式

在关键节点输出当前状态：

```markdown
📊 当前步骤：步骤 X - [步骤名称]
📁 工作目录：./codespec/changes/{work_dir}/

文档状态：
- ✅ proposal.md（已完成，时间戳）
- ⏳ delta-spec.md（进行中，问答进度 3/5）
- ⏸️ delta-design.md（待开始）
- ⏸️ tasks.md（待开始）
- ⏸️ validation.md（待开始）

当前任务：[具体任务描述]
```

## 重入恢复逻辑

### 重入检测

当 skill 被重新激活时，执行以下检测：

```
1. 检查对话历史中的最后状态信息
2. 扫描 ./codespec/changes/ 目录，查找最近的工作目录
3. 检查工作目录下已存在的文档
4. 识别当前处于哪个步骤
```

### 重入确认

向用户确认当前状态：

```markdown
🔍 检测到您正在进行 SDD 设计

工作目录：./codespec/changes/US202601010015-增加逻辑多租
当前步骤：步骤 2 - 增量规格定义（问答阶段）

已完成的文档：
- ✅ proposal.md（2026-02-11 14:30）

当前任务：
- 正在进行 delta-spec.md 的问答澄清
- 已覆盖 3/5 个维度

是否继续？请选择：
- [ ] 继续当前步骤
- [ ] 跳转到其他步骤
- [ ] 开始新的设计
```

### 状态恢复

根据用户选择执行相应的恢复操作：

**继续当前步骤**：
- 加载当前步骤的问答进度
- 询问未覆盖的维度

**跳转到其他步骤**：
- 验证目标步骤的前置条件
- 加载目标步骤的文档（如果存在）
- 询问用户是修订还是重新生成

**开始新的设计**：
- 执行步骤 0：工作目录识别
- 重置所有状态变量

## 交互式澄清原则

### 信息控制

- **单次提问量**：每次不超过 3 个问题
- **回复长度**：单次回复控制在 30 行以内
- **信息密度**：聚焦关键信息，避免冗余

### 动态调整

- **根据用户反馈**：如果用户回答简短，追问细节；如果用户回答详细，减少追问
- **根据复杂度**：简单需求快速推进，复杂需求分阶段深入
- **根据上下文**：利用已收集的信息，避免重复询问

### 渐进式披露

- **从宏观到微观**：先问背景和目标，再问细节和边界
- **从核心到边缘**：先问核心功能，再问异常场景和边界条件
- **从需求到约束**：先问功能需求，再问技术约束和非功能需求

## 异常处理

### 用户回答不完整

**检测**：用户回答部分问题或跳过某些问题

**处理**：
```markdown
我注意到您没有回答关于 [问题主题] 的问题。

这个信息对设计很重要，请问：
- [ ] 现在补充回答
- [ ] 暂时跳过，后续补充
- [ ] 使用默认值：[默认值说明]
```

### 用户回答冲突

**检测**：当前回答与之前的回答矛盾

**处理**：
```markdown
⚠️ 检测到回答冲突

您之前提到：
- "[之前的回答]"

现在又说：
- "[现在的回答]"

请问以哪个为准？
- [ ] 之前的回答
- [ ] 现在的回答
- [ ] 两者都是（请说明场景）
```

### 用户请求调整

**检测**：用户说"不对"、"我改变主意了"等

**处理**：
```markdown
收到，需要调整之前的回答。

请告诉我：
1. 哪个部分需要调整？
   - [ ] 业务价值
   - [ ] 功能范围
   - [ ] 用户故事
   - [ ] 验收标准

2. 新的理解是什么？
```

## 输出格式规范

### 问答输出格式

```markdown
📊 当前步骤：步骤 X - [步骤名称]
🎯 问答进度：进行中（已覆盖维度数/总维度数）

❓ 请回答以下问题：

1. [问题1]
2. [问题2]
3. [问题3]

💡 提示：[可选的提示信息]
```

### 确认输出格式

```markdown
✅ 收到，感谢您的回答！

📝 我理解的是：

- [要点1]
- [要点2]
- [要点3]

请确认理解是否正确：
- [ ] 正确，继续下一步
- [ ] 需要修正
```

### 进度总结格式

```markdown
📊 问答进度总结

✅ 已完成维度：
- 业务价值：[总结]
- 目标用户：[总结]

⏳ 待确认维度：
- 功能范围
- 验收标准

🎯 下一步：继续澄清功能范围
```
