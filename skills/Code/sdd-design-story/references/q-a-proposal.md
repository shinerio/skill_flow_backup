# 步骤 1：需求澄清的问答逻辑

本文档提供 proposal.md 生成过程中的关键问题集和流程逻辑，用于引导用户提供必要的需求信息。

> **通用引导技巧**：详见 [./common-guidance.md](./common-guidance.md)

---

## 🚨 阶段边界提醒

**当前阶段：步骤1 - 需求澄清（Proposal）**

**本阶段聚焦**：
- ✅ 为什么要做（Why）- 业务背景、动机、价值
- ✅ 做什么（What Changes）- 功能范围、边界、用户故事
- ✅ 谁使用、怎么算完成 - 目标用户、验收标准

**本阶段禁止**：
- ❌ 技术实现细节（如算法、数据类型、存储方式）→ 步骤3（Design）
- ❌ 数据模型设计（如表结构、字段定义）→ 步骤3（Design）
- ❌ 具体代码结构（如类名、方法签名）→ 步骤4（Tasks）

**技术细节将在步骤3的Design阶段讨论，代码实现在步骤4的Tasks阶段讨论**

## 场景识别

在开始问答之前，先识别 story 的类型，选择合适的问答策略。

### 场景类型

**1. 业务需求场景**
- **特征**：从零开始的新需求，需要明确业务价值和目标用户
- **示例**：新增用户管理功能、新增支付接口、新增报表功能
- **问答策略**：从"为什么做"开始，聚焦业务价值和用户需求

**2. 设计分解场景**
- **特征**：来自前期设计/架构决策的分解，目标已经明确
- **示例**：初始化服务项目框架、实现数据访问层、实现 API 网关
- **问答策略**：跳过"为什么做"，聚焦"如何实现"和"验收标准"

**3. 技术改进场景**
- **特征**：技术优化、重构、性能改进，目标明确
- **示例**：优化查询性能、重构认证模块、升级依赖版本
- **问答策略**：聚焦技术方案和改进目标

### 场景识别逻辑

**第一轮识别问题**（仅 1-2 个问题）：
```markdown
📊 当前步骤：步骤 1 - 需求澄清

请先简要描述这个 story 的来源和目标：

1. 这个 story 来自哪里？
   - [ ] 新的业务需求
   - [ ] 前期设计/架构决策的分解
   - [ ] 技术优化/重构
   - [ ] 其他（请说明）

2. 这个 story 的核心目标是什么？（一句话）
```

**根据场景选择问答策略**：
- 如果选择"业务需求" → 使用【业务需求场景】问答逻辑
- 如果选择"设计分解"或"技术改进" → 使用【设计分解场景】问答逻辑

## 问答流程逻辑

### 业务需求场景

#### 初始阶段（第一轮）

**目标**：获取核心背景信息

**必问问题**：
1. **业务背景**
   - "这个 story 要解决什么业务问题？"
   - "当前的解决方案是什么？有什么不足？"

2. **目标用户**
   - "谁是这个 story 的目标用户？"
   - "用户的核心痛点是什么？"

**输出格式**：
```markdown
📊 当前步骤：步骤 1 - 需求澄清
🎯 问答进度：初始阶段（0/4 维度已覆盖）

❓ 请回答以下问题：

1. 这个 story 要解决什么业务问题？当前的解决方案是什么？
2. 谁是这个 story 的目标用户？用户的核心痛点是什么？
```

---

### 设计分解场景

#### 初始阶段（第一轮）

**目标**：明确实施范围和技术约束

**必问问题**（聚焦 How，而非 Why）：

**注意**：对于设计分解场景，假设"为什么做"已经明确，直接聚焦"如何实现"。

1. **核心功能**（聚焦具体实现）
   - "这个 story 具体要实现什么功能？"
   - "核心交付物是什么？（如：代码文件、配置文件、脚本等）"

2. **技术约束**（聚焦技术实现）
   - "基于什么技术栈和框架？"
   - "需要遵循哪些编码规范或架构标准？"
   - "有哪些技术限制或外部依赖？"

**输出格式**：
```markdown
📊 当前步骤：步骤 1 - 需求澄清（设计分解场景）
🎯 问答进度：初始阶段（0/4 维度已覆盖）

❓ 请回答以下问题（聚焦技术实现）：

1. 这个 story 具体要实现什么功能？核心交付物是什么？
2. 基于什么技术栈？需要遵循哪些规范？
```

**示例输出**（针对"初始化服务项目框架"）：
```markdown
📊 当前步骤：步骤 1 - 需求澄清（设计分解场景）
🎯 问答进度：初始阶段（0/4 维度已覆盖）

❓ 请回答以下问题（聚焦技术实现）：

1. 这个 story 具体要实现什么功能？核心交付物是什么？
   - 例如：创建 Spring Boot 项目骨架、配置文件结构、基础依赖配置等

2. 基于什么技术栈？需要遵循哪些规范？
   - 例如：Spring Boot 3.x + MyBatis，遵循项目编码规范和目录结构规范
```

#### 深化阶段（第二轮）

**目标**：明确实施细节和验收标准

**问题选择策略**：

**如果涉及数据操作** → 询问**数据范围**
- "涉及哪些业务数据？"
- "数据之间有什么业务关联？"

**如果涉及接口设计** → 询问**功能范围**
- "需要提供哪些功能接口？"
- "接口的核心功能是什么？"

**如果涉及配置/框架** → 询问**配置需求**
- "需要配置哪些业务参数？"
- "配置的业务含义是什么？"

**默认深化问题**（如果没有触发特定场景）：
1. **实施范围**
   - "具体要实现哪些模块/组件？"
   - "有哪些功能不在范围内？"

2. **验收标准**
   - "怎样算做完了？如何验证？"
   - "有哪些可量化的指标？"

**输出格式**：
```markdown
📊 当前步骤：步骤 1 - 需求澄清（设计分解场景）
🎯 问答进度：深化阶段（2/4 维度已覆盖）

✅ 已覆盖维度：
- 核心功能
- 技术约束

❓ 请回答以下问题：

1. 具体要实现哪些模块/组件？有哪些功能不在范围内？
2. 怎样算做完了？如何验证？
```

#### 确认阶段（最后轮）

**目标**：总结理解，确认边界和依赖

**必问问题**：
1. **依赖确认**
   - "是否有依赖的其他功能或系统？"
   - "需要前置条件吗？"

2. **影响范围**
   - "会影响哪些现有功能？"
   - "是否有破坏性变更？"

**理解确认**：
```markdown
📝 我理解的是：

**核心功能**：
- [总结用户的回答]

**技术约束**：
- [总结技术栈和规范]

**实施范围**：
- [总结要实现的模块/组件]
- [明确排除的功能]

请确认理解是否正确：
- [ ] 正确，继续下一步
- [ ] 需要修正
```

---

### 业务需求场景 - 深化阶段（第二轮）

**目标**：根据第一轮回答，深入关键维度

**问题选择策略**：

**如果用户提到"性能问题"** → 询问**性能要求**
- "预期的用户量是多少？"
- "响应时间要求是多少？"

**如果用户提到"多用户/并发"** → 询问**并发量**
- "预期的并发请求量是多少？"
- "是否有高峰期？如何应对？"

**如果用户提到"数据安全/隐私"** → 询问**安全要求**
- "需要什么级别的认证？"
- "有哪些敏感数据需要保护？"

**如果用户提到"新旧系统切换"** → 询问**迁移方案**
- "是否有破坏性变更？"
- "如何处理历史数据？"

**默认深化问题**（如果没有触发特定场景）：
1. **功能范围**
   - "具体要做什么？不做什么？边界在哪里？"
   - "核心功能有哪些？"

2. **用户故事**
   - "作为[角色]，我希望[做什么]，以便[获得什么价值]？"
   - "能否举一个具体的使用场景？"

**输出格式**：
```markdown
📊 当前步骤：步骤 1 - 需求澄清
🎯 问答进度：深化阶段（2/4 维度已覆盖）

✅ 已覆盖维度：
- 业务价值
- 目标用户

❓ 请回答以下问题：

1. 具体要做什么？不做什么？边界在哪里？
2. 能否描述一个典型的使用场景？
```

---

**目标**：总结理解，确认边界和验收标准

**必问问题**：
1. **边界确认**
   - "有哪些功能不在范围内？"
   - "是否有依赖的其他功能或系统？"

2. **验收标准**
   - "怎样算做完了？如何验证？"
   - "有哪些可量化的指标？"

**理解确认**：
```markdown
📝 我理解的是：

**业务价值**：
- [总结用户的回答]

**目标用户**：
- [总结用户的回答]

**功能范围**：
- [总结核心功能]
- [明确排除的功能]

请确认理解是否正确：
- [ ] 正确，继续下一步
- [ ] 需要修正
```

## 问题库

### 业务价值

- 这个需求解决什么痛点？带来什么价值？
- 这个 story 要解决什么业务问题？
- 当前的解决方案是什么？有什么不足？
- 这个 story 的优先级是什么？为什么是现在做？

### 功能范围

- 具体要做什么？不做什么？边界在哪里？
- 核心功能有哪些？
- 功能之间的依赖关系是什么？
- 有哪些功能是必须的，哪些是锦上添花的？

### 用户故事

- 谁是这个 story 的目标用户？
- 谁会使用？使用场景是什么？
- 用户的核心痛点是什么？
- 用户期望获得什么价值？

### 验收标准

- 怎样算做完了？如何验证？
- 如何判断功能已经完成？
- 有哪些可量化的指标？
- 需要进行哪些测试？

## 进度检查点

在生成 proposal.md 之前，检查以下内容：

### ✅ 必须覆盖的维度

- [ ] 业务价值（为什么要做）
- [ ] 目标用户（谁使用）
- [ ] 功能范围（做什么/不做什么）
- [ ] 验收标准（怎么算完成）

### ⏸️ 可选维度（根据需求）

- [ ] 性能要求
- [ ] 安全要求
- [ ] 兼容性要求
- [ ] 技术约束

### 📋 生成前确认

如果缺少必选维度，询问用户：
```markdown
📋 生成 proposal.md 前的最后确认

我发现以下信息还不够完整：
- [ ] 验收标准：怎样算做完了？

是否：
- [ ] 现在补充
- [ ] 使用默认推断
- [ ] 暂时跳过（在文档中标注"待确认"）
```

## 重入恢复逻辑

当 skill 在步骤 1 被重新激活时：

1. **检查已覆盖维度**
   - 查找对话历史中的问答进度
   - 识别已覆盖的维度

2. **确认理解**
   ```markdown
   📊 检测到您正在进行步骤 1 - 需求澄清

   ✅ 已覆盖维度：
   - 业务价值：[总结]
   - 目标用户：[总结]

   ⏳ 待覆盖维度：
   - 功能范围
   - 验收标准

   我理解的是...，对吗？
   ```

3. **继续问答**
   - 询问未覆盖的维度
   - 根据用户反馈调整

## 边界自检

在进入下一轮问答前，执行以下边界自检：

### 自检清单

- [ ] 是否讨论了技术实现细节？（如算法、数据类型、存储方式）
  - 如果是 → 标记为"待Design阶段讨论"，继续聚焦业务需求
- [ ] 是否涉及数据模型设计？（如表结构、字段定义）
  - 如果是 → 标记为"待Design阶段讨论"，继续聚焦数据范围
- [ ] 是否涉及具体代码结构？（如类名、方法签名）
  - 如果是 → 标记为"待Tasks阶段讨论"，继续聚焦功能范围

### 边界提醒话术

如果用户开始讨论技术细节，使用以下话术：

```markdown
💡 提醒：技术实现细节属于 Design 阶段

当前我们正在讨论 Proposal（需求澄清），技术实现细节（如算法、数据类型、存储方式）会在步骤 3 的 Design 阶段讨论。

如果您想讨论技术方案，我们可以：
- [ ] 先完成 Proposal 阶段，再进入 Design 阶段
- [ ] 跳转到步骤 3 的 Design 阶段
```

### 阶段完成标准

在进入步骤 2（增量规格）之前，确保：

- [ ] 已明确业务背景和动机
- [ ] 已明确目标用户和核心痛点
- [ ] 已明确功能范围和边界
- [ ] 已定义验收标准
- [ ] 未涉及技术实现细节
- [ ] 未涉及数据模型设计
- [ ] 未涉及具体代码结构
