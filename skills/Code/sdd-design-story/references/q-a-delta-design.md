# 步骤 3：增量设计的问答逻辑

本文档提供 delta-design.md 生成过程中的引导问题集和流程逻辑，用于深化设计细节。

**前置条件**：已读取全量设计文档（`{root}/codespec/specs/design.md`）

> **通用引导技巧**：详见 [./common-guidance.md](./common-guidance.md)

---

## 🚨 阶段边界提醒

**当前阶段：步骤3 - 增量设计（Design）**

**本阶段聚焦**：
- ✅ 技术方案是什么（How）- 技术选型、方案对比
- ✅ 数据设计 - 数据存储策略、数据结构组织
- ✅ 接口设计 - 接口技术规范、数据格式
- ✅ 架构设计 - 模块依赖、影响范围
- ✅ 权衡取舍 - 风险评估、缓解方案

**本阶段禁止**：
- ❌ 具体代码实现（如类名、方法签名、设计模式）→ 步骤4（Tasks）
- ❌ 数据库DDL语句（如CREATE TABLE、ALTER TABLE）→ 步骤4（Tasks）
- ❌ 数据迁移脚本（如迁移SQL、回滚脚本）→ 步骤4（Tasks）
- ❌ 具体的框架使用细节（如Spring Boot配置、注解使用）→ 步骤4（Tasks）

**代码实现细节将在步骤4的Tasks阶段讨论**

## 场景识别

在开始问答之前，识别 story 的类型，选择合适的问答策略。

### 场景类型

**1. 业务需求场景**
- **特征**：新增业务功能，需要设计决策
- **示例**：新增用户管理、新增支付功能、新增报表功能
- **问答策略**：询问技术方案、方案对比、数据模型、接口设计

**2. 设计分解场景**
- **特征**：技术方案已在 proposal 或 delta-spec 中明确，需要澄清细节
- **示例**：实现数据访问层、实现 API 网关、实现消息队列
- **问答策略**：跳过"技术方案选择"，聚焦具体实现细节和技术约束

**3. 技术改进场景**
- **特征**：技术优化、重构，技术方案固定
- **示例**：优化查询性能、重构认证模块、升级依赖版本
- **问答策略**：聚焦改进措施、性能优化、风险缓解

### 场景识别逻辑

**判断依据**（基于步骤 1 的场景识别）：
- 如果步骤 1 识别为"业务需求" → 使用【业务需求场景】问答逻辑
- 如果步骤 1 识别为"设计分解" → 使用【设计分解场景】问答逻辑
- 如果步骤 1 识别为"技术改进" → 使用【技术改进场景】问答逻辑

## 问答流程逻辑

### 业务需求场景

#### 初始阶段（第一轮）

**目标**：识别设计变更范围

**必问问题**：
1. **设计变更识别**
   - "基于 delta-spec.md，需要变更哪些设计模块？"
   - "是新增模块、修改现有模块，还是删除模块？"

2. **技术选型概览**
   - "倾向使用什么技术方案？"
   - "有哪些备选方案？"

**输出格式**：
```markdown
📊 当前步骤：步骤 3 - 增量设计（业务需求场景）
🎯 问答进度：初始阶段（0/5 维度已覆盖）

📋 基于 delta-spec.md 的设计变更识别：
- [ ] 新增模块
- [ ] 修改现有模块
- [ ] 删除模块

❓ 请回答以下问题：

1. 主要的设计变更点是什么？涉及哪些模块？
2. 倾向使用什么技术方案？有哪些备选方案？
```

#### 深化阶段（第二轮）

**目标**：深入技术细节和数据模型

**问题选择策略**：

**如果有数据模型变更** → 询问**数据设计**
- "数据结构如何设计？与现有模型如何关联？"
- "数据库表结构如何变更？需要新增/修改哪些表？"
- "索引如何优化？"

**如果有接口变更** → 询问**接口设计**
- "API 如何设计？参数和返回值是什么？"
- "接口的向后兼容性如何保证？"

**如果有复杂业务流程** → 询问**流程设计**
- "业务流程如何设计？"
- "状态机如何定义？"
- "异常流程如何处理？"

**默认深化问题**：
1. **架构设计**
   - "架构变更的影响范围是什么？"
   - "模块之间的依赖关系如何？"

2. **权衡取舍**
   - "性能 vs 可维护性？如何权衡？"
   - "性能瓶颈在哪里？有哪些优化措施？"

**输出格式**：
```markdown
📊 当前步骤：步骤 3 - 增量设计（业务需求场景）
🎯 问答进度：深化阶段（2/5 维度已覆盖）

✅ 已覆盖维度：
- 设计变更识别
- 技术选型概览

❓ 请回答以下问题：

1. 数据结构如何设计？与现有模型如何关联？
2. 架构变更的影响范围是什么？模块之间的依赖关系如何？
```

#### 确认阶段（最后轮）

**目标**：确认设计方案和风险评估

**必问问题**：
1. **方案确认**
   - "最终选择的技术方案是什么？"
   - "选择这个方案的理由是什么？"

2. **风险评估**
   - "有哪些技术风险？如何缓解？"
   - "需要预研的技术点有哪些？"

**理解确认**：
```markdown
📝 我理解的设计方案：

**数据模型变更**：
- [总结数据模型]

**接口设计**：
- [总结接口设计]

**架构设计**：
- [总结架构设计]

**权衡取舍**：
- [总结权衡取舍]

请确认理解是否正确：
- [ ] 正确，继续下一步
- [ ] 需要修正
```

### 设计分解场景

#### 初始阶段（第一轮）

**目标**：澄清具体实现细节

**注意**：假设技术方案已在 proposal 或 delta-spec 中明确，跳过"技术方案选择"的问题。

**必问问题**（聚焦实现，而非选择）：
1. **实现细节**
   - "proposal 或 delta-spec 中提到的技术方案，有哪些实现细节需要澄清？"
   - "具体的技术选型是什么？"

2. **技术约束**
   - "需要遵循哪些技术规范或标准？"
   - "有哪些技术限制或依赖？"

**输出格式**：
```markdown

❓ 请回答以下问题（聚焦实现细节）：

1. proposal 或 delta-spec 中提到的技术方案，有哪些实现细节需要澄清？
2. 需要遵循哪些技术规范或标准？有哪些技术限制或依赖？
```

#### 深化阶段（第二轮）

**目标**：深入数据模型和接口规范

**问题选择策略**：

**如果涉及数据操作** → 询问**数据设计**
- "数据存储策略是什么？"
- "数据结构如何组织？"
- "如何保证数据性能？"

**如果涉及接口设计** → 询问**接口设计**

- "接口的数据格式是什么？"
- "错误处理策略是什么？"

- "配置管理策略是什么？"
- "配置的加载机制是什么？"

**默认深化问题**：
1. **实现范围**
   - "具体要实现哪些模块/组件？"
   - "有哪些功能不在范围内？"

2. **技术依赖**
   - "需要依赖哪些外部系统或库？"
   - "版本要求是什么？"

```markdown

2. 具体要实现哪些模块/组件？需要依赖哪些外部系统？
```

## 问题库

### 技术选型

- 倾向使用什么技术方案？

- 方案对比的依据是什么？
- 选择这个方案的理由是什么？
- 是否有预研的技术点？

- 与现有模型如何关联？

- 性能瓶颈在哪里？
- 有哪些优化措施？

- 有哪些技术风险？
- 如何缓解这些风险？

## Design vs Implementation 边界检查

在问答过程中，始终注意避免进入 Implementation 领域：

### ❌ 不应该问的问题（属于 Implementation）

- "具体使用哪个类来实现？"（Implementation）
- "方法的签名是什么？"（Implementation）
- "使用什么设计模式？"（Implementation）
- "代码结构如何组织？"（Implementation）

### ✅ 应该问的问题（属于 Design）

- "模块的职责是什么？"（Design）
- "模块之间的接口是什么？"（Design）
- "数据如何流转？"（Design）
- "如何保证数据一致性？"（Design）

### 边界提醒

如果用户开始讨论代码实现，提醒：
```markdown
💡 提醒：代码实现细节属于 Implementation 阶段

当前我们正在讨论 Design（技术方案），代码实现细节（如类名、方法签名、设计模式）会在步骤 4 的 Tasks 阶段讨论。

如果您想讨论代码实现，我们可以：
- [ ] 先完成 Design 阶段，再进入 Implementation 阶段
- [ ] 跳转到步骤 4 的 Tasks 阶段
```
